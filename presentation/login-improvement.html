Stage 1

localStorage UUID

🔗 bang-tang.com

실제 로그인이 없다. 앱 최초 실행 시 브라우저에 랜덤 UUID를 생성해 저장하고, 이것을 익명 세션 ID로 활용한다. 서버도, 계정도 없다.

src/lib/session.ts — 2026-04-03 · c012ea4
const SESSION_KEY = 'escape-room-session'

/** 앱 최초 실행 시 생성되는 익명 UUID */
export function getSessionId(): string {
  let id = localStorage.getItem(SESSION_KEY)
  if (!id) {
    id = crypto.randomUUID()
    localStorage.setItem(SESSION_KEY, id)
  }
  return id
}
Chrome DevTools · Application → Local Storage

장점

  • 구현이 매우 간단하다
  • 서버나 외부 서비스가 필요 없다
  • 앱 실행 즉시 동작한다
  • 네트워크 비용이 없다

단점

  • 기기를 바꾸면 데이터가 사라진다
  • 브라우저 초기화 시 UUID가 소멸한다
  • "내 계정"이 존재하지 않는다
  • 개인화·보안이 불가능하다

Stage 1 · localStorage → Stage 2 · Supabase Auth

개선 이유

기록이 "내 것"이어야 했다

📱 스마트폰으로 방탈출 기록을 쌓았다가 PC로 접속하면 기록이 없어졌다
👤 UUID는 누가 사용하는지 알 수 없어 개인화된 서비스를 제공할 수 없었다
☁️ 방탈출 기록을 서버에 저장해 어떤 기기에서도 이어서 볼 수 있어야 했다
🔐 실제 계정을 부여해 나중에 보안 정책을 적용할 수 있는 기반을 만들고 싶었다
Stage 2

Supabase Auth — 이메일 매직링크

🔗 bang-tang.com 📊 Supabase Dashboard

이메일을 입력하면 로그인 링크가 발송되고, 클릭하는 순간 로그인된다 (OTP). Supabase 하나로 인증과 DB를 함께 관리한다.

src/components/UserAuthButton.tsx — 2026-04-15 · d171c6c
// 이메일 OTP 로그인 전송
async function sendMagicLink(e: FormEvent) {
  e.preventDefault()
  const { error } = await supabase.auth.signInWithOtp({
    email: email.trim(),
    options: {
      emailRedirectTo: window.location.origin,
      shouldCreateUser: true,
    },
  })
  if (error) { setStatus('error'); return }
  setStatus('sent') // "메일함에서 링크를 확인해 주세요"
}

// 세션 감지 → 로그인 성공 시 모달 닫기
supabase.auth.onAuthStateChange((_event, session) => {
  setSession(session)
  if (session) setOpen(false)
})

장점

  • 실제 사용자 계정이 생긴다
  • 기기 간 데이터 동기화가 가능하다
  • 비밀번호 없이 로그인할 수 있다

단점

  • 로그인 UI·에러 처리를 직접 구현해야 한다
  • 이메일 도착을 기다리는 UX 마찰이 있다
  • 인증과 DB가 Supabase에 강결합된다

Stage 2 · Supabase Auth → Stage 3 · Clerk + Supabase

개선 이유

인증은 전문가에게 맡기자

🎨 이메일 입력 폼·에러 메시지·모달을 직접 만들고 유지하는 게 번거로웠다
🌐 소셜 로그인(Google, GitHub 등)을 추가하려면 인증 코드가 급격히 복잡해진다
🔗 인증과 DB가 Supabase에 뭉쳐 있어 각자의 역할을 분리하기 어려웠다
✨ Clerk는 완성도 높은 UI와 소셜 로그인을 제공해 구현 비용을 크게 줄일 수 있었다
Stage 3

Clerk + Supabase — 역할 분리

🔗 bang-tang.com 👤 Clerk Dashboard

Clerk가 인증(UI 포함)을, Supabase가 데이터를 담당한다. Clerk에서 발급된 JWT를 Supabase에 전달해 RLS 정책이 그대로 작동한다.

src/lib/auth.tsx + supabaseClient.ts — 2026-04-24 · 04aa4c3
// auth.tsx — Clerk JWT를 Supabase에 주입
function ClerkAuthSync({ children }) {
  const { isLoaded, getToken } = useClerkAuth()

  useEffect(() => {
    if (!isLoaded) return
    setSupabaseAccessTokenGetter(async () => {
      const token = await getToken() // Clerk JWT
      return token ?? null
    })
  }, [isLoaded, getToken])

  return <>{children}</>
}

// supabaseClient.ts — 요청마다 토큰 자동 포함
export const supabase = createClient(url, anonKey, {
  accessToken: async () =>
    accessTokenGetter ? accessTokenGetter() : null,
})
Clerk 로그인 UI (기본 제공 · 소셜 로그인 포함)
Clerk 유저 대시보드

장점

  • Clerk가 제공하는 완성도 높은 로그인 UI
  • 소셜 로그인을 설정만으로 추가한다
  • 인증(Clerk)과 데이터(Supabase) 역할이 분리된다

단점

  • 두 가지 외부 서비스에 의존하게 된다
  • Clerk → Supabase JWT 연동 설정이 필요하다
  • 규모에 따라 Clerk 유료 플랜이 필요할 수 있다
